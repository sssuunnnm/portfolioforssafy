GPS 노이즈를 어떻게 길들였나

버튼을 눌러 산책 한 번을 시뮬레이션해 보세요. 같은 데이터, 다른 결과.

문제: 산책 중 GPS가 한 번 튀면 → 폴리라인이 어지럽게 그려지고, 거리가 부풀어 오르고, 심지어 자동 도착 판정이 잘못 발동해 산책이 끝나버렸습니다. 도메인 계층과 ViewModel에 4중 가드를 넣어 해결했어요.
📡 Raw GPS (FusedLocationProvider 원본) UNFILTERED
수신: 0 총거리: 0.0m 도착 오판: 0회
✅ 4중 가드 통과 FILTERED
채택: 0 총거리: 0.0m 도착 판정: 대기
목적지에 도달하면 필터링 쪽만 정상 종료합니다.
▎ 적용된 가드 (WalkViewModel.kt & TrackLocationUseCase.kt 실값)
accuracy < 20f — TrackLocationUseCase: GPS 정확도 20m 이상 폐기 (도메인 계층 필터)
distance < 100m — 이전 포인트와 100m 이상 점프 시 누적에서 제외 (텔레포트 방어)
ARRIVE_MAX_ACCURACY 10f — 도착 판정은 더 엄격, 정확도 10m 이하만 인정
ARRIVE_STABLE_COUNT 3 — 연속 3회 임계치 안에 들어와야 도착 확정
DESTINATION_THRESHOLD 20.0m — 목적지 반경 20m 이내
ARRIVE_MIN_ELAPSED_MS 3000ms — 시작 직후 GPS 안정화 시간 보장

// WalkViewModel.kt — 별도 상태를 들지 않고, 두 흐름에서 파생
val averageSpeed: StateFlow<Double> = combine(
    totalDistance, elapsedTime
) { distance, time ->
    if (time > 0) (distance / 1000.0) / (time / 3600.0) else 0.0
}.stateIn(viewModelScope, SharingStarted.Lazily, 0.0)
타이머가 매 초 +1, 거리가 임의 보폭으로 증가 → averageSpeed가 자동으로 따라옵니다.
