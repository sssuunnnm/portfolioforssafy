규제 문서 처리 추적(SSE)과 토큰 갱신(race condition) 두 모듈을, 레포의 실제 코드 값과 분기를 그대로 시각화했습니다. 다이어그램의 상태를 클릭하거나 시뮬레이션을 실행해 흐름을 따라가 보세요.

1

SSE 연결 상태 머신

src/features/regulation/hooks/useRegulationProcessingTracker.ts

단순 구독이 아니라 스냅샷 선조회 → 진행 중일 때만 스트림 연결 → 비정상 종료 시 자동 재연결까지 처리하는 상태 머신입니다. 각 상태를 클릭하면 해당 분기를 담당하는 실제 코드가 표시됩니다.

상태 전이 다이어그램

idle syncing streaming completed failed retry
// 재생 시 SSE 이벤트가 여기에 출력됩니다
useRegulationProcessingTracker.ts
설계 포인트. EventSource는 JWT 헤더를 못 실어 fetch + ReadableStream으로 직접 파싱하고, 문서별 AbortController·재시도 타이머를 Map으로 관리합니다. 완료/실패 시에만 localStorage를 정리하고 목록 쿼리를 무효화해, 새 탭·새로고침에도 추적이 복원됩니다.
2

동시 401 → 단 한 번의 토큰 갱신

src/api/client.ts

요청 3개가 동시에 401을 맞아도 토큰 갱신은 한 번만. 첫 요청이 '갱신 담당'이 되고 나머지는 subscribe()로 큐에 대기하다가, 갱신이 끝나면 notify()로 한꺼번에 깨워 재시도합니다.

공유 플래그
isRefreshing = false
refreshAccessToken()
만료된 토큰 갱신 횟수 0
subscriber 큐 subscribers[]
// 비어 있음
0
client.ts · 응답 인터셉터 (401 분기)
설계 포인트. isRefreshing 플래그 + subscribers[] 큐로 중복 갱신을 막습니다. _retry 플래그로 무한 루프를 차단하고, refresh 요청 자체의 실패는 별도 분기에서 즉시 forceLogout() 처리합니다.
